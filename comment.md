# Virtual DOM 발표용 정리

이 문서는 현재 프로젝트를 기준으로, **Virtual DOM의 원리를 발표에서 어떻게 설명하면 이해가 잘 되는지**를 정리한 발표 준비용 메모다.  
목표는 "코드를 전부 설명하는 것"이 아니라, **왜 Virtual DOM이 필요한지, 실제로 어떤 순서로 동작하는지, 이 프로젝트에서 그 원리가 어떻게 보이는지**를 자연스럽게 전달하는 것이다.

---

## 1. 발표의 핵심 메시지

발표에서 가장 먼저 잡아야 할 메시지는 아래 한 줄이다.

**Virtual DOM은 화면 전체를 매번 다시 바꾸는 대신, 이전 상태와 다음 상태를 비교해서 필요한 변경만 실제 DOM에 반영하려는 아이디어다.**

이 한 줄이 발표 전체의 중심축이 되면 좋다.

즉 발표는 아래 순서로 흘러가면 이해가 쉽다.

1. 왜 필요한가
2. 어떻게 동작하는가
3. 이 프로젝트에서 그것이 어디에 보이는가
4. 결과적으로 어떤 장점이 있는가

---

## 2. 발표를 시작할 때 좋은 도입

처음부터 코드 설명으로 들어가면 듣는 사람이 바로 피곤해질 수 있다.  
그래서 먼저 "문제 상황"으로 시작하는 것이 좋다.

예시 멘트:

> 웹 화면에서 어떤 값 하나가 바뀌었을 때, 가장 단순한 방법은 화면 전체를 다시 그리는 것입니다.  
> 하지만 실제로는 바뀐 부분만 수정하는 것이 더 효율적입니다.  
> Virtual DOM은 바로 이 지점을 해결하기 위한 아이디어입니다.

또는 더 짧게:

> 사용자가 버튼 하나를 눌렀다고 해서 화면 전체를 다시 만들 필요는 없습니다.  
> 우리가 진짜 알고 싶은 것은 "정확히 어디가 바뀌었는가"입니다.

이렇게 시작하면 이후 Diff, Patch 설명으로 자연스럽게 넘어갈 수 있다.

---

## 3. 발표 전체 흐름 추천

가장 설명이 잘 먹히는 발표 구조는 아래다.

### 1단계. Virtual DOM이 왜 필요한가

설명 포인트:

- 실제 DOM은 브라우저가 직접 관리하는 화면 구조다.
- 실제 DOM을 자주 직접 건드리면 변화 추적이 복잡해진다.
- 화면이 복잡해질수록 "무엇이 바뀌었는지"를 사람이 직접 관리하기 어려워진다.
- 그래서 실제 DOM을 직접 비교하는 대신, 비교하기 쉬운 **가벼운 JS 객체 트리**를 하나 더 둔다.

예시 멘트:

> Virtual DOM은 진짜 화면 그 자체가 아니라, 화면을 설명하는 가벼운 설계도 같은 객체 구조입니다.

### 2단계. Virtual DOM이 어떻게 동작하는가

이 부분은 반드시 아래 4단어로 설명하는 것이 좋다.

- **Create**
- **Compare**
- **Diff**
- **Patch**

예시 멘트:

> 먼저 현재 화면을 Virtual DOM 형태로 표현합니다.  
> 그리고 사용자가 수정한 다음 상태도 다시 Virtual DOM으로 만듭니다.  
> 그 둘을 비교해서 차이(diff)를 구하고, 마지막에 필요한 부분만 실제 DOM에 반영합니다.

### 3단계. 이 프로젝트에서 어디에 보이는가

이 프로젝트 기준으로 연결하면 설명이 훨씬 살아난다.

- `domToVdom()`이 실제 DOM을 VDOM으로 바꾼다.
- `diff()`가 이전 트리와 다음 트리를 비교한다.
- `applyPatches()`가 필요한 부분만 실제 DOM에 반영한다.
- `history`가 상태를 저장해서 undo/redo를 가능하게 한다.

예시 멘트:

> 이 프로젝트에서는 테스트 영역을 수정하면, 그 DOM을 다시 VDOM으로 바꾸고, 이전 상태와 비교해서 patch 목록을 만듭니다.  
> 그리고 Patch 버튼을 누르면 그 결과만 실제 화면에 반영됩니다.

### 4단계. 결과적으로 무엇이 좋은가

설명 포인트:

- 바뀐 부분만 반영할 수 있다.
- 변경 내용을 추적하기 쉽다.
- undo/redo 같은 상태 복원이 쉬워진다.
- 로그, 시각화, 트리 비교 같은 디버깅도 쉬워진다.

예시 멘트:

> Virtual DOM의 장점은 단순히 빠르다는 것보다, 변화의 단위를 명확하게 다룰 수 있다는 데 있습니다.

---

## 4. 현재 프로젝트 기준으로 설명하기 좋은 핵심 흐름

이 프로젝트에서 Virtual DOM 원리를 보여주려면 아래 흐름만 제대로 설명해도 충분하다.

### 1. 초기 상태 만들기

- 샘플 보드를 실제 DOM으로 만든다.
- 그 DOM을 `initialTree`라는 VDOM으로 바꾼다.
- 같은 트리를 `testRoot`에도 렌더링한다.

설명 멘트:

> 처음에는 실제 DOM에서 출발하지만, 곧바로 그것을 Virtual DOM 트리로 변환해서 기준 상태를 만듭니다.

### 2. 사용자가 테스트 영역을 수정함

- 버튼 클릭
- 온도 입력
- 방 카드 reorder
- 카드 테마 토글

설명 멘트:

> 사용자가 수정하는 곳은 실제 반영 영역이 아니라 실험용 테스트 영역입니다.

### 3. 수정된 테스트 DOM을 다시 VDOM으로 변환함

- `domToVdom(testRoot.firstElementChild)`

설명 멘트:

> 수정이 끝나면, 방금 바뀐 DOM을 다시 Virtual DOM 트리로 바꿉니다.  
> 즉, 이전 상태도 트리, 다음 상태도 트리로 만들어서 비교 가능한 형태로 맞추는 것입니다.

### 4. 이전 트리와 다음 트리를 비교함

- `diff(store.getTree(), nextTree)`

설명 멘트:

> 여기서 핵심은 화면을 다시 그리는 것이 아니라, 두 트리의 차이만 계산한다는 점입니다.

### 5. patch를 실제 DOM에 반영함

- `applyPatches(currentActualNode, patches)`

설명 멘트:

> 마지막에는 계산된 patch만 실제 DOM에 반영합니다.  
> 그래서 전체를 갈아엎지 않고, 바뀐 부분만 수정하는 구조가 됩니다.

## 5. Virtual DOM 원리를 설명할 때 꼭 강조하면 좋은 개념

### 1. 실제 DOM과 Virtual DOM은 다르다

이 부분은 꼭 분리해서 말하는 것이 좋다.

- 실제 DOM: 브라우저가 화면에 그리는 진짜 구조
- Virtual DOM: 그 화면을 JS 객체로 표현한 비교용 구조

짧은 멘트:

> 실제 DOM은 결과물이고, Virtual DOM은 비교를 위한 중간 표현입니다.

### 2. 핵심은 "전체 재렌더"가 아니라 "차이 계산"이다

많은 사람이 Virtual DOM을 그냥 "화면 다시 그리는 기술"로 이해하는데, 발표에서는 이 오해를 바로잡아주는 것이 좋다.

짧은 멘트:

> Virtual DOM의 핵심은 다시 그리는 것이 아니라, 무엇이 달라졌는지 계산하는 데 있습니다.

### 3. Diff가 있어야 Patch가 가능하다

설명 포인트:

- Diff는 차이를 찾는 단계
- Patch는 찾은 차이를 실제 DOM에 반영하는 단계

짧은 멘트:

> Diff는 분석이고, Patch는 실행입니다.

### 4. 상태를 트리로 보면 undo/redo도 쉬워진다

이 프로젝트에 특히 잘 맞는 포인트다.

짧은 멘트:

> 상태를 DOM 자체가 아니라 트리 스냅샷으로 저장하니까, 이전 상태로 돌아가는 것도 쉬워집니다.

## 6. 발표에서 보여주기 좋은 데모 순서

이 프로젝트 화면을 직접 보여주면서 설명할 때는 아래 순서가 좋다.

### 데모 1. 전등 ON/OFF

왜 좋은가:

- 가장 직관적이다.
- `PROPS`와 `TEXT`가 같이 보일 수 있다.

멘트:

> 전등을 ON에서 OFF로 바꾸면, 전체 카드가 바뀌는 것이 아니라 상태 관련 속성과 텍스트만 바뀝니다.

### 데모 2. 에어컨 온도 변경

왜 좋은가:

- input `value`와 readout text가 같이 바뀐다.
- "속성 변화 + 텍스트 변화"를 설명하기 좋다.

멘트:

> 숫자를 바꾸면 input의 value와 화면에 보이는 텍스트가 바뀝니다.  
> Virtual DOM은 이런 세부 변화도 따로 추적할 수 있습니다.

### 데모 3. 방 카드 순서 변경

왜 좋은가:

- keyed diff와 `MOVE` 설명에 좋다.
- Virtual DOM을 더 깊게 이해했다는 인상을 준다.

멘트:

> 이 경우는 카드가 사라졌다가 새로 생긴 것이 아니라, 같은 카드가 위치만 이동한 것입니다.  
> 그래서 key를 기반으로 같은 노드를 식별해서 `MOVE`로 처리합니다.

### 데모 4. 카드 테마 토글

왜 좋은가:

- 클래스와 `data-theme`, 버튼 텍스트가 함께 바뀐다.
- props와 text 차이를 같이 보여주기 좋다.

멘트:

> 테마를 바꾸면 카드 전체를 새로 만드는 것이 아니라, 필요한 속성과 텍스트만 바뀝니다.

### 데모 5. Undo / Redo

왜 좋은가:

- 상태 저장과 복원의 장점을 보여준다.

멘트:

> 이렇게 계산된 상태를 트리 형태로 저장해 두면, undo/redo도 단순히 이전 트리를 복원하는 방식으로 동작할 수 있습니다.

## 7. 발표에서 Diff를 쉽게 설명하는 방법

Diff를 너무 어렵게 설명하면 듣는 사람이 바로 놓친다.  
그래서 처음에는 아래 3가지로만 설명하는 것이 좋다.

- **새로 생기면 `CREATE`**
- **사라지면 `REMOVE`**
- **같은 노드인데 속성만 바뀌면 `PROPS`**

그리고 여유가 있으면 다음을 추가한다.

- text만 바뀌면 `TEXT`
- 위치만 바뀌면 `MOVE`
- 노드 종류가 달라지면 `REPLACE`

예시 멘트:

> Diff는 결국 "새로 생겼나, 사라졌나, 같은데 내용만 바뀌었나"를 판별하는 과정입니다.

또는:

> 처음에는 복잡해 보이지만, 실제로는 변화의 종류를 몇 가지로 나눠서 보는 방식입니다.

## 8. 발표에서 자주 나오는 질문에 대한 답변 준비

### 질문 1. Virtual DOM이 실제 DOM보다 왜 좋은가요?

답변 포인트:

- 실제 DOM은 브라우저가 직접 관리하므로 다루기 복잡하다.
- Virtual DOM은 비교하기 쉬운 JS 객체 구조다.
- 중요한 것은 "중간에서 차이를 계산할 수 있다"는 점이다.

짧은 답:

> 실제 DOM을 직접 비교하는 대신, 비교하기 쉬운 객체 구조를 한 번 거치면 변경 추적이 쉬워집니다.

### 질문 2. Virtual DOM이면 항상 빠른가요?

답변 포인트:

- 무조건 빠르다고 말하면 위험하다.
- 핵심은 성능 하나보다도 **변경 관리가 쉬워진다**는 점이다.

짧은 답:

> 핵심은 무조건 빠르다기보다, 변화의 단위를 명확하게 관리할 수 있다는 데 있습니다.

### 질문 3. 이 프로젝트에서 Virtual DOM의 핵심 코드는 어디인가요?

답변:

- `domToVdom()`
- `diff()`
- `applyPatches()`

짧은 답:

> 이 프로젝트에서는 DOM을 VDOM으로 바꾸는 `domToVdom`, 차이를 찾는 `diff`, 실제 적용을 담당하는 `applyPatches`가 핵심입니다.

### 질문 4. 왜 test 영역과 actual 영역을 나눴나요?

답변 포인트:

- 비교 실험을 명확하게 보여주기 위해서다.
- 사용자가 수정한 상태와 실제 반영된 상태를 분리해서 볼 수 있다.

짧은 답:

> 수정 중인 상태와 실제 반영된 상태를 나눠 보여줘야 Diff와 Patch 과정을 눈으로 설명하기 쉽기 때문입니다.

## 9. 발표할 때 추천하는 3분 버전 구성

시간이 짧다면 아래 구조가 가장 무난하다.

### 1분

- Virtual DOM이 왜 필요한지 설명
- 실제 DOM을 직접 다루는 대신 비교용 트리를 둔다고 설명

### 1분

- 이 프로젝트 흐름 설명
- `DOM -> VDOM -> diff -> patch -> DOM 반영`

### 1분

- 데모 1개 또는 2개
- ON/OFF 변경 + Undo/Redo 정도만 보여줘도 충분

짧은 마무리 멘트:

> 정리하면 Virtual DOM은 화면을 객체 트리로 표현하고, 이전 상태와 다음 상태의 차이만 계산해서 실제 DOM에 반영하는 방식입니다.

## 10. 발표할 때 추천하는 5분 버전 구성

시간이 5분 정도면 아래처럼 하면 좋다.

### 1분

- 문제 제기: 화면 전체 재렌더 vs 필요한 부분만 변경

### 1분

- Virtual DOM 개념 설명
- 실제 DOM과 Virtual DOM 차이 설명

### 1분

- 프로젝트 구조 설명
- `domToVdom`, `diff`, `applyPatches`, `history`

### 1분

- 데모: 전등 상태 변경, 온도 변경

### 1분

- 데모: 카드 reorder 또는 undo/redo
- 마무리 요약

## 11. 슬라이드 한 장으로 정리할 때 좋은 문장

슬라이드용으로는 아래 문장이 가장 쓰기 좋다.

### 짧은 버전

**Virtual DOM은 화면을 객체 트리로 표현하고, 이전 상태와 다음 상태를 비교해 필요한 변경만 실제 DOM에 반영하는 방식이다.**

### 프로젝트 맞춤 버전

**이 프로젝트는 테스트 영역의 DOM을 Virtual DOM으로 변환하고, 이전 트리와 diff를 계산한 뒤, patch만 실제 화면에 반영하는 구조로 Virtual DOM 원리를 보여준다.**

### 상태 관리 강조 버전

**Virtual DOM은 단순 렌더링 기술이 아니라, 변화 추적과 상태 복원을 쉽게 만들어 주는 중간 표현이다.**

## 12. 발표 마무리 멘트 예시

마지막 20초 정도는 아래처럼 정리하면 좋다.

> 이번 프로젝트를 통해 Virtual DOM의 핵심은 "가상으로 한 번 더 그린다"가 아니라,  
> "이전 상태와 다음 상태를 비교해서 바뀐 부분만 실제 DOM에 반영한다"는 점이라는 것을 확인할 수 있었습니다.  
> 그리고 이 방식은 화면 업데이트뿐 아니라, patch 로그, 시각화, undo/redo 같은 기능으로도 확장될 수 있습니다.

더 짧게 끝내고 싶다면:

> 정리하면 Virtual DOM은 화면을 객체 트리로 바꾸고, 차이만 계산해서 실제 화면에 반영하는 아이디어입니다.
